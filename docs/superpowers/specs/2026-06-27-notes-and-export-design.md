# お知らせ／あとがき（前書き・後書き）と投稿サイト向けエクスポート — 設計

> 作成日: 2026-06-27
> 対象: noveditor（core=Kotlin/KMP、web=TS+Vite+Svelte5）
> 優先順: **Phase 1（お知らせ/あとがき）を先行**、続いて **Phase 2（記法対応エクスポート）**。
> 方針: calm / offline-first を維持。本文を汚さない構造化フィールド方式。

## 背景・調査結果（投稿サイト連携／2026-06 時点）

- なろう・カクヨム・アルファポリスとも **作品投稿の公式 API は無い**。投稿はフォーム手動（カクヨムは貼付のみ、なろう/アルファは 1 話ずつ .txt アップロード可＝△）。
- 各社の「エクスポート」は **バックアップ目的の .txt(ZIP) ダウンロード**で、他サイトへ戻すインポート口は公式に無い。
- **記法が各社で異なる**（傍点：カクヨム/アルファ=`《《…》》`、なろうは `｜《《…》》` 等にエスケープ／ルビ区切りも差）。
- 結論：**自動投稿は不可**（やるならスクレイピング＝ToS リスクで非推奨）。現実的かつ安全なのは **記法に変換したテキストのエクスポート**（貼付/1話アップロード用）と、**.txt/ZIP のインポート**。
- 出典: なろうヘルプ「投稿済み作品テキストDL」、カクヨムヘルプ「小説をバックアップする」、Ryuon「各サイトのDL/UP対応一覧」「ルビ・傍点記法の違い」。

# Phase 1: お知らせ／あとがき（前書き・後書き）

なろう/カクヨムの「まえがき・あとがき」に相当。**話ごと**と**小説共通（全話）**の両方を持つ。空なら自動で隠し、「＋」で追加（使わない人の画面は素のまま）。呼び名は **お知らせ（冒頭）／あとがき（末尾）**。

## ① データモデル（core / Kotlin → JS → TS）

既存コンストラクタを壊さないよう、**末尾に既定値付きフィールドを追加**する（既存呼び出しはそのまま動く）。

- `Episode(id, novelId, title, body, createdAt, updatedAt, foreNote = "", afterNote = "")`
- `Novel(id, title, synopsis, episodeOrder, createdAt, updatedAt, foreNote = "", afterNote = "")`
- `EpisodeSummary` / `NovelSummary` は変更しない（一覧に注記は不要）。
- JS export（`Exports.kt` / `.d.mts`）と web 側 DTO / 型ミラーに同フィールドを追加。

**移行**: localStorage の既存 DTO に新フィールドが無い場合は `""` を補う（`fromEpisodeDto`/`fromNovelDto`）。バリデータ（`isEpisodeDto`/`isNovelDto`）は新フィールドを**必須にしない**。＝追加的・冪等・後方互換（破壊的移行なし）。

**文字数**: 原稿用紙換算・セッション字数・各カウントは **本文（body）のみ**。お知らせ/あとがきは数えない。

## ② AppState（web/src/state/appState.svelte.ts）

- 編集バッファに `foreNote = $state('')`, `afterNote = $state('')`（現在話のお知らせ/あとがき）を追加。
- `loadEpisodeInto` で `ep.foreNote`/`ep.afterNote` を読み込み、`saveNow` で `new Episode(..., now, this.foreNote, this.afterNote)` として保存。`markEdited` の対象に含める。
- 小説共通は**小説メタ**として扱う：`novelForeNote`/`novelAfterNote` を保持し、`getNovelMeta`/`updateNovelMeta` を title/synopsis に加えて fore/after も扱うよう拡張。`touchNovel`（保存時の updatedAt 更新）が既存の fore/after を保持するよう注意。

## ③ エディタ UI（話ごと）— web/src/editor/Editor.svelte

- 既定は本文のみ。**タイトル下に「＋お知らせ」**、**本文下に「＋あとがき」**の控えめなリンク。
- 押すと該当欄（本文より小さめ・淡色の小テキストエリア）が現れ編集可。**空にして離れると自動で畳む**（次回は「＋」に戻る）。
- 既に内容がある話は最初から欄を表示。
- 自動保存は本文と同じ経路（`onEdited`/`markEdited`）に統合。IME 対応（compositionstart/end）も本文同様。

## ④ 小説共通のお知らせ／あとがき — web/src/list/NovelList.svelte（小説情報の編集）

- 既存の「小説情報を編集」（タイトル・あらすじ）に「小説のお知らせ」「小説のあとがき」欄を追加。空運用は同じ。
- ラベルは level を明示：話側＝「お知らせ／あとがき」、小説側＝「小説のお知らせ／小説のあとがき」。

## ⑤ プレビュー／読み（web/src/editor/Preview.svelte）

表示順（**空でなければ**それぞれ表示）:

1. 小説のお知らせ → 2. この話のお知らせ → 3. **本文** → 4. この話のあとがき → 5. 小説のあとがき

- 注記は本文と区別する控えめな体裁（小さめ・淡色・細い罫 or 淡いタイント枠）。ルビ/圏点も本文同様に `renderRuby` を通す。
- Preview は現在 `body`/`title` のみ受け取る。`foreNote`/`afterNote`（話・小説）も props で渡す。

## ⑥ テスト

- core: Episode/Novel の新フィールド（既定値・往復）jvmTest。
- web: repository 往復（fore/after の save→load→list、欠落 DTO の補完＝移行）。countStats が本文のみであることの確認は既存維持。
- 既存 32 件を緑のまま維持。`pnpm build` 緑。

## ⑦ スコープ外（Phase 1）

- エクスポート/インポート本体（Phase 2）。
- 縦書き（tatemd 連携時に同構造を渡す）。

# Phase 2: 記法対応エクスポート（お知らせ/あとがき実装後に詳細設計）

> 高価値・オフライン・ToS 安全。Phase 1 のデータ（本文・ルビ・圏点・お知らせ/あとがき）を各社記法に変換して出力。**自動投稿はしない**（貼付 or 1話ずつ .txt アップロード用）。

## 出力プロファイル（要・実装時に最終確認）

| 項目 | なろう | カクヨム | アルファポリス |
|---|---|---|---|
| ルビ | `｜親文字《ふりがな》` | `｜親文字《ふりがな》` | （要確認） |
| 傍点 | `｜《《…》》` 等のエスケープ | `《《…》》` | `《《…》》` |
| 前書き/後書き | エピソード前書き/後書き欄へ | 同左 | 同左 |
| 出力単位 | 話ごと .txt（1話ずつUP） | 貼付（UP非対応） | 話ごと .txt |

- noveditor 内部表記（`｜漢字《かんじ》`＝ルビ、`《《…》》`＝圏点）→ 各プロファイルへ変換する純関数を `core`（commonMain）か web util に実装。**記法差は個別に裏取りしてから**実装（調査の出典を参照）。
- UI: 「エクスポート」から サイト/単位を選び、**話ごとにコピー or .txt ダウンロード（必要に応じ ZIP）**。

## Phase 2 のインポート（任意・移行用）

- .txt / 各社バックアップ ZIP を話に分割して取り込み。各社のファイル構成差（カクヨムの about.txt 等）を個別処理。

## 慎重に進める点（Phase 2）

- 公式 API 無し＝自動投稿しない。スクレイピング/自動投稿は ToS リスクのため不採用。
- 記法（特に傍点エスケープ）とルビ区切りの差を**プロファイル単位で実データ確認**してから変換実装。
- バックアップ ZIP 形式は各社で異なるため、インポートは形式ごとに分けて対応。
